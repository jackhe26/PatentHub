## 三、核心技术分析与Bug修复（重点）

### 3.1 原有初始化逻辑分析

```typescript
// 旧的逻辑（有Bug）
const initializePatentCopilots = async (currentCopilots) => {
  const patentCopilotIds = patentCopilots.map(p => p.id)  // 新9个搭子的ID
  const existingIds = currentCopilots.map(c => c.id)       // 已存储搭子的ID

  // 检查：已有任一搭子吗？
  const hasPatentCopilots = patentCopilotIds.some(id => existingIds.includes(id))

  if (!hasPatentCopilots) {
    // 首次启动 → 添加所有新搭子
    return [...currentCopilots, ...patentCopilots]
  }

  // 非首次启动 → 直接返回旧数据，不做任何更新！
  return currentCopilots
}
```

### 3.2 问题复现现象
> **用户反馈：** "显示的还是之前默认的8个审查搭档"

修改了 `patent-copilots.ts` 文件，但打开应用后依然显示旧的8个搭子。

### 3.3 根因分析

**场景1：ID匹配的旧搭子 → 索引被保留**
```
旧ID: patent-copilot:01-invention → 新ID: patent-copilot:01-invention  ✅ matched
结果：旧搭子被保留（旧prompt内容不变）
```

**场景2：ID不匹配的旧搭子 → 作为自定义搭子残留**
```
旧: patent-copilot:02-claim-cgk       → 新ID列表中没有这个ID → 旧搭子残留！
旧: patent-copilot:03-translation     → 新ID列表中没有这个ID → 旧搭子残留！
旧: patent-copilot:04-search-expression → 新ID列表中没有这个ID → 旧搭子残留！
旧: patent-copilot:06-unity            → 新ID列表中无此ID → 旧搭子残留！
旧: patent-copilot:08-creative-review  → 新ID列表中无此ID → 旧搭子残留！
```

**根源：** 由于旧搭子的 `id` 并不完全在新9个搭子的 `id` 列表中，导致：

1. **`hasPatentCopilots = true`**（因为 `01-invention` 等部分ID匹配）
2. 直接执行 `return currentCopilots` → **旧数据原封不动返回**
3. 旧搭子（ID不匹配的）继续保留在存储中，未清除
4. UI 依然显示旧搭子

### 3.4 解决思路

**放弃基于ID匹配的判断逻辑**，改用**ID前缀识别**策略：

```typescript
// 所有专利搭档的ID都以该前缀开头
const PATENT_COPILOT_ID_PREFIX = 'patent-copilot:'

const initializePatentCopilots = async (currentCopilots) => {
  // 检查是否有任何专利搭子（通过前缀识别）
  const hasPatentCopilots = currentCopilots.some(c => 
    c.id.startsWith(PATENT_COPILOT_ID_PREFIX)
  )

  if (!hasPatentCopilots) {
    // 首次初始化：添加所有新搭子
    return [...currentCopilots, ...patentCopilots]
  }

  // ★ 核心修复：过滤掉所有旧专利搭子（通过前缀），替换为最新数据
  // 无论旧搭子ID是否在新列表中，全部清除
  const nonPatentCopilots = currentCopilots.filter(
    c => !c.id.startsWith(PATENT_COPILOT_ID_PREFIX)
  )
  return [...nonPatentCopilots, ...patentCopilots]
}
```

### 3.5 修复效果

| 对比维度 | 修复前 | 修复后 |
|---------|--------|--------|
| 判断机制 | 精确ID匹配 | 前缀识别 |
| 首次启动 | ✅ 正常添加 | ✅ 正常添加 |
| 新增搭子时 | ❌ 旧数据残留 | ✅ 全量替换为最新 |
| 删除搭子时 | ❌ 旧搭子残留 | ✅ 自动清除 |
| 搭子数量变化时 | ❌ 不做检测 | ✅ 自动匹配源数据 |
| 自定义搭子 | ✅ 不受影响 | ✅ 不受影响（保留非 `patent-copilot:` 前缀搭子） |

## 六、经验教训与最佳实践

### 6.1 幂等初始化设计的陷阱
- **"首次检测"** 策略只适用于**永不修改源数据的场景**
- 一旦源数据可能变化（增/删/改），必须要有**版本检测**或**全量替换**机制

### 6.2 本地持久化缓存的最佳实践

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 首次检测（旧方案） | 性能好，不重复写入 | 源数据变更时不更新 | 固定不变的初始数据 |
| 前缀+全量替换（新方案） | 始终与源数据同步 | 每次启动都写入 | 源数据可能被修改 |
| 版本号检测 | 按需更新，性能最佳 | 需要维护版本号 | 大型复杂系统 |

### 6.3 修改静态数据文件的注意事项

1. **检查引用链路**：修改了数据源后，必须检查所有引用该数据的组件和Hook
2. **关注持久化缓存**：`atomWithStorage` / `localStorage` 会缓存旧数据，需要主动刷新
3. **运行时验证**：不仅要看编译是否通过，还要在运行时验证实际渲染的数据
4. **ID前缀约定**：使用统一的前缀（如 `patent-copilot:`）可以在逻辑层灵活识别同类数据
