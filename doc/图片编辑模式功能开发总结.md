# 图片编辑模式（参考图上传）功能开发总结

## 概述

为 PatentHub 的图片生成功能增加 **编辑模式（Edit Mode）** 支持：用户上传参考图片或从历史记录中选择参考图后，通过 `POST /v1/images/edits` API 提交编辑请求，实现基于参考图的图片生成。

## 改动文件（4 个）

| 文件 | 改动类型 | 说明 |
|------|---------|------|
| `src/shared/models/imageEditApi.ts` | **新建** | 封装 `POST /v1/images/edits` 的 multipart 请求核心逻辑 |
| `src/shared/models/abstract-ai-sdk.ts` | 修改 | 基类增加编辑模式方法 + 自动回退机制 |
| `src/shared/providers/definitions/models/custom-openai.ts` | 修改 | 自定义 OpenAI 提供商覆写编辑模式参数 |
| `src/shared/providers/definitions/models/chatboxai.ts` | 修改 | ChatboxAI 提供商覆写编辑模式参数 |

## 架构设计

### 编辑模式数据流

```
用户上传参考图 / 选历史图
  → index.tsx: handleSubmit()
    → createAndGenerate({ referenceImages, size })
      → imageGenerationActions.ts: generateImages()
        → model.paint({ images, size })
          ├─ 📄 images=[], 无参考图 → 文生图（原有路径）
          └─ 📄 images.length>0, 有参考图 → 编辑模式

编辑模式内部：
  paintWithEdits()
    ├─ 尝试当前模型: POST {baseUrl}/v1/images/edits (multipart)
    │   ├─ dataUrlToBlob() × 图片数量
    │   ├─ buildEditsFormData(image, prompt, model, n, size)
    │   └─ parseEditResponse() → b64_json | url(下载转base64)
    │
    └─ 失败 → 自动回退 gpt-image-2 模型重试
         (同 baseUrl + apiKey，仅改 model 字段)
```

### 子类覆写方法

基类 `AbstractAISDKModel` 定义了三个 `protected` 方法供子类覆写：

```typescript
protected getEditModeBaseUrl(): string | undefined  // 编辑 API 的基础 URL
protected getApiKey(): string | undefined            // 编辑 API 的认证密钥
protected getEditModeModelId(): string               // 编辑使用的模型 ID
```

各提供商的实现：

| 提供商 | baseUrl | apiKey | modelId |
|--------|---------|--------|---------|
| Custom OpenAI | `apiHost.replace(/\/v1\/?$/, '')` | `apiKey` | `model.modelId` |
| ChatboxAI（非Gemini） | `${getChatboxAPIOrigin()}/gateway/openai/v1` | `licenseKey` | `model.modelId` |
| ChatboxAI（Gemini） | ❌ 不走编辑模式，走原有 `paintWithGemini()` | - | - |

## 关键设计决策

### 1. 同步模式选择

Python 参考代码中有两种模式：
- **同步**：`POST /v1/images/edits` → 直接返回图片数据
- **异步**：`POST /v1/images/edits?async=true` → 返回 task_id → 轮询 `/v1/images/tasks/{task_id}`

选择 **同步模式**，因为：
- 代码更简洁，不需要管理任务状态和轮询
- Electron 桌面应用不需要异步回调
- 后续可扩展异步模式

### 2. 图片尺寸的确定

编辑模式完全复用已有的 **比例 + 分辨率 UI 选择器**：

```
selectedRatio(16:9) + selectedResolution(2k)
  → getImageSize("16:9", "2k") = "2560x1440"
  → 传给 edit API 的 size 字段
  → API 用此尺寸生成输出图片
```

**`size` 控制的是输出图片尺寸**，与输入参考图的实际分辨率无关。多张输入图尺寸不一致时，由 API 自行处理适配。

### 3. 图片来源的统一

| 来源 | 存储格式 | 传递给 model 的格式 |
|------|---------|-------------------|
| 本地上传 | `storage.setBlob(key, dataUrl)` | `storage.getImage(key)` → `{ imageUrl: dataUrl }` |
| 历史参考图 | 同上 | 同上 |

两者在提交时统一为 `{ imageUrl: string }[]`，编辑 API 层无需区分来源。

### 4. 自动回退机制

编辑模式下按以下顺序尝试模型：

1. **当前用户选择的模型**（如 `gpt-image-1.5`）
2. **失败后自动回退** → `gpt-image-2`

回退使用**同一个 baseUrl 和 apiKey**，仅修改 `model` 字段。这种方式适配了 API 网关模式（同一网关代理多个模型）。

## 经验教训

### Lesson 1: API Host 路径归一化

**问题**：首次请求报 404，URL 为 `https://ai.comfly.chat/v1/v1/images/edits`

**根因**：`custom-openai.ts` 的 `getEditModeBaseUrl()` 直接返回 `this.options.apiHost`（值为 `https://ai.comfly.chat/v1`），而 `editWithEditsAPI()` 拼接了 `/v1/images/edits`，导致双 `/v1/`。

**修复**：
```typescript
// custom-openai.ts - 去除 apiHost 末尾的 /v1
return this.options.apiHost.replace(/\/v1\/?$/, '')

// imageEditApi.ts - 智能拼接路径
const endpoint = normalizedBase.endsWith('/v1')
  ? `${normalizedBase}/images/edits`
  : `${normalizedBase}/v1/images/edits`
```

**启示**：API Host 的路径构成因提供商而异，编码时必须考虑：
- 有的包含 `/v1`（Comfly: `https://ai.comfly.chat/v1`）
- 有的不包含（OpenAI: `https://api.openai.com`）
- 不能假设格式统一，要做归一化处理

### Lesson 2: API 响应格式多样性

**问题**：生成的图片无法显示，控制台报 `ERR_INVALID_URL`，data URL 变成了 `data:image/png;base64,https://oss.filenest.top/uploads/...`

**根因**：API 返回的是图片 URL（`https://oss.filenest.top/uploads/...`）而非 `b64_json`。`parseEditResponse()` 直接把 URL 原样返回，下游代码检测到不以 `data:` 开头就粗暴包裹 `data:image/png;base64,` 前缀。

**修复**：在 `parseEditResponse()` 中增加 `downloadUrlAsDataUrl()` 函数，当 API 返回 `url` 时主动下载并转为 base64。

```typescript
async function downloadUrlAsDataUrl(url: string): Promise<string> {
  const response = await fetch(url)
  const blob = await response.blob()
  const arrayBuffer = await blob.arrayBuffer()
  const base64 = btoa(new Uint8Array(arrayBuffer)...)
  return `data:${blob.type};base64,${base64}`
}
```

**启示**：图片生成 API 的响应格式不统一，主要有三种情况：
- `{ b64_json: "base64..." }` — 直接 base64 数据
- `{ url: "https://..." }` — 可下载的图片 URL
- `{ url: "data:image/png;base64,..." }` — 完整 data URL

代码需要同时处理这三种情况。

### Lesson 3: 多层继承的设计

`AbstractAISDKModel` 作为基类，`CustomOpenAI` 和 `ChatboxAI` 作为子类。编辑模式的核心逻辑放在基类，子类只覆写参数获取方法。这种设计的好处：
- 减少重复代码
- 统一编辑/回退行为
- 子类只需关注配置注入

### Lesson 4: 保留原有逻辑路径

ChatboxAI 的 Gemini 模型走原有的 `paintWithGemini()`（Google 原生多模态输入），非 Gemini 模型走编辑 API。这是重要的兼容性处理，避免破坏已有功能。

## 测试要点

1. **本地上传单张参考图** → 编辑生成 ✅
2. **历史记录选参考图** → 编辑生成 ✅
3. **多张参考图** → 编辑生成（API 自行适配）✅
4. **无参考图** → 文生图（原有路径不受影响）✅
5. **尺寸选择** → `auto` 不传 size / 选择比例+分辨率传对应尺寸 ✅
6. **模型回退** → 当前模型失败自动用 gpt-image-2 重试 ✅
7. **Gemini 模型** → 不走编辑模式，保持原有行为 ✅
