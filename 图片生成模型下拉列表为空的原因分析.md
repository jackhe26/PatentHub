# 图片生成模型下拉列表为空的原因分析和修复

## 问题描述

在 `ImageCreatorPage`（图片生成页面）中，点击模型选择下拉框时，除了默认显示的 "GPT Image" 外，没有其他第三方自定义提供商的图片模型列表显示。但在聊天界面的模型选择器中，同一提供商的模型却能正常显示。

## 根因分析

### 数据流链路

```
用户配置的自定义提供商 (type=openai, isCustom=true)
    ↓
useProviders() 返回 providers 列表
    ↓
ImageModelSelect.tsx 中 customOpenAIProviders 过滤逻辑
    ├─ 条件1: p.isCustom === true ✅
    ├─ 条件2: p.type === 'openai' ✅
    └─ 条件3: m.capabilities?.includes('image') ❌ ← 这里出问题
```

### 核心 Bug

**`capabilities` 字段缓存丢失**。原因链如下：

1. Zod Schema 的 `.catch([])` 默认值机制：

   ```typescript
   // src/shared/types/settings.ts - ProviderModelInfoSchema
   capabilities: z
     .array(z.enum(['vision', 'reasoning', 'tool_use', 'web_search', 'image']))
     .optional()
     .catch([])   // ← 如果数据格式不对，静默降级为空数组
   ```

2. 模型列表数据保存在 `localStorage` 中，经过序列化/反序列化后，`capabilities` 字段可能因为各种原因（数据版本变化、手动编辑、schema 校验失败等）被 Zod 的 `.catch([])` 重置为空数组。

3. `ImageModelSelect.tsx` 的过滤逻辑**单独依赖** `capabilities.includes('image')`：

   ```typescript
   .filter((m) => m.capabilities?.includes('image'))
   // 当 capabilities 为空数组 → 所有模型都被过滤 → 下拉框为空
   ```

4. 聊天界面使用的模型选择器**不需要 `capabilities` 过滤**，所有模型全部展示 → 永远正常。

### 为什么"删除模型 ID → 重新加载"就好了

- 删除模型 ID 后，系统会重新调用 `GET /v1/models` 拉取最新模型列表。
- 重新拉取时，代码会重新执行模型名称 → capabilities 的映射逻辑，正确打上标签。
- 所以 capabilities 数据被刷新后，过滤条件通过，下拉框恢复正常。

## 修复方案

### 修改文件 1：`src/renderer/components/ImageModelSelect.tsx`

在 `customOpenAIProviders` 的模型过滤逻辑中，**增加名称关键词后备匹配**：

```typescript
.filter((m) => {
  // Primary check: explicit 'image' capability tag
  if (m.capabilities?.includes('image')) return true
  // Fallback: match by model name keywords (handles cached data where capabilities may be lost)
  const name = (m.nickname || m.modelId || '').toLowerCase()
  return ['image', 'sd', 'flux', 'banana', 'dall-e', 'gpt-image', 'paint', 'draw'].some(
    (keyword) => name.includes(keyword)
  )
})
```

### 修改文件 2：`src/renderer/routes/image-creator/index.tsx`

同样的后备匹配逻辑同步到 `imageModelGroups` 中（移动端/小屏模式也受此影响）。

## 经验教训

1. **不要单独依赖动态数据的单字段进行过滤**。`capabilities` 是可缓存、可序列化的数据，任何环节的 Schema 校验失败或数据版本变化都可能导致字段值意外清空。

2. **Zod 的 `.catch()` 默认值是一个静默陷阱**。它会让数据"看起来正常"但内容为空，不会抛出任何错误或警告，给调试带来极大困难。

3. **UI 过滤逻辑应该设计为"宽进严出"**。对于展示类过滤（如"哪些模型适合图片生成"），在主要检查（capabilities）失效时，应该有后备匹配机制（如名称关键词匹配）作为降级方案。

4. **同一功能的多个组件间过滤逻辑要保持一致**。`ImageModelSelect.tsx`（桌面版下拉框）和 `index.tsx` 的 `imageModelGroups`（移动版抽屉）的过滤条件不一致，导致行为不同步。

5. **隐性 Bug 的特征**：
   - 不会稳定复现（有时好有时坏）
   - 依赖数据状态而非代码逻辑
   - 用户无意的操作（如手动编辑模型列表、切换页面等）可能触发或修复
   - 不同入口（聊天/图片生成）行为不一致
